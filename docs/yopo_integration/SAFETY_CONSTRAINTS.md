# YOPO 实机集成安全约束

## 1. 总原则

任何阶段的“能运行”都不等于“可飞行”。安全门禁 fail closed：缺少输入、来源冲突、
时间过期、frame 不符、数值无效、配置 hash 不符或诊断不健康时，禁止继续向下游
发布新的授权数据。

## 2. 永久禁止项

- 软件自动解锁；
- 节点重启后自动恢复 OFFBOARD；
- 飞行中自动切换 cuVSLAM 与动捕；
- 在同一 epoch 内重置 `map` 或持续重新对齐；
- 多个 publisher 同时拥有 PX4 external vision 或控制话题；
- 在 ROS 侧手工预转换 NED/FRD 后再交给 MAVROS；
- 将缺失速度伪造成有效零速度；
- 未经无桨门禁直接装桨执行新控制链；
- 直接控制电机；
- 首版执行高速、窄缝、移动障碍或人员附近飞行。

## 3. 人工权限

- 飞手和遥控器拥有最高权限；
- 解锁与进入 OFFBOARD 必须是明确人工动作；
- 遥控模式切换必须能绕过 YOPO 和 SO3；
- 测试前必须确认遥控 failsafe、PX4 OFFBOARD loss 行为和急停流程；
- 任何操作者无法解释的状态都必须中止测试。

## 4. 首版飞行包络

| 约束 | 上限 |
| --- | --- |
| 最大倾角 | 15 deg |
| 最大偏航角速度 | 30 deg/s |
| 最大水平速度 | 0.5 m/s |
| 最大垂直速度 | 0.3 m/s |
| 最大加速度 | 1.0 m/s^2 |
| 最高高度 | 2.5 m |
| 单次目标距离 | 3.0 m |
| 静态障碍最小规划安全间距 | 0.5 m |

飞行器完整起飞重量为 0.689 kg，X 布局，包含机圈的两条对角外轮廓均为
248 mm。场地边界必须额外包含定位误差、制动距离和飞手缓冲区，不能只按机体
124 mm 半径计算。

## 5. 故障处理

### 5.1 深度或 YOPO 轨迹失效

前提是定位仍健康：

1. 停止生成新的 YOPO 轨迹；
2. SO3 不得无限重放最后一条轨迹；
3. 转入经过台架验证的安全保持路径；
4. 请求 PX4 `AUTO.LOITER` 或由飞手接管。

### 5.2 SO3 或控制 setpoint 失效

- 停止发布新的姿态目标；
- 依赖已明确配置并测试的 PX4 OFFBOARD failsafe；
- 禁止软件在后台自动重新进入 OFFBOARD。

### 5.3 主定位失效

- 立即撤销主定位 gateway 授权；
- 禁止自动切换到影子定位源；
- 退出 OFFBOARD 并要求飞手接管；
- 只有高度、姿态和下降条件仍可信且对应流程已验证时才允许自动降落。

### 5.4 定位跳变或 epoch 变化

定位重启、GID 改变、时间回退、Pose reset 或固定对齐变化均结束当前 epoch。
飞行器必须落地后才能建立新 epoch。

## 6. 阶段门禁

顺序不可跳过：

1. 单元、lint 和接口合同测试；
2. 深度与定位联合运行；
3. YOPO 被动规划；
4. 拆桨控制联调和故障注入；
5. `mocap_primary` 低速飞行；
6. `cuvslam_primary` 空场飞行；
7. 静态障碍避障。

前一阶段证据没有标记 `PASSED` 时，下一阶段保持 `BLOCKED`。

## 7. emitter 策略

首轮深度和 cuVSLAM 联合测试保持红外发射器关闭。如果深度有效率不足，只允许在
独立 A/B 测试中开启，并重新验证：

- 两路红外图像频率与时间戳；
- cuVSLAM tracking 状态和 odometry 频率；
- 深度有效像素比例和空洞分布；
- Jetson 温度、内存和 GPU 负载。

不得因为深度改善而默认接受 cuVSLAM 回归。

## 8. 证据与复现

每次装桨测试前必须生成只读清单，至少包含：

- 所有相关仓库 commit hash；
- PX4 固件版本和关键参数导出；
- MAVROS 版本；
- launch/config SHA-256；
- mode 和 localization epoch 标识；
- 前置测试报告；
- 飞手、观察员、场地和日期。

任何与清单不一致的运行都视为新的未验证配置。
